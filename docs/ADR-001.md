# Architecture Decision Record: Orchestration, Model Selection, and Prompt Design

* **Status**: Accepted
* **Date**: 2026-06-02

---

## 1. Context & Problem Statement

The goal is to build a robust, end-to-end Document Intelligence Pipeline MVP that ingests a PDF, extracts structured JSON data, publishes it, and verifies its accuracy against the source text. 

The main architectural decisions to address are:
1. **Model Stack Selection**: Running with Anthropic API vs. Google Gemini API.
2. **Orchestration**: Selecting n8n vs. Trigger.dev (or utilizing a lightweight CLI orchestration script for the MVP).
3. **Prompt Design & Schema Validation**: Ensuring high-quality JSON extraction and structured verification.

---

## 2. Decision & Justification

### 2.1 Model Stack Selection: Anthropic API (Claude 3.5 Haiku)
In compliance with the assignment brief stack requirement, we migrated the model selection to the Anthropic API.
* **Decision**: We opted for the official `anthropic` SDK, running **Claude 3.5 Haiku** (`claude-3-5-haiku-20241022`) by default, with option to configure **Claude 3.5 Sonnet** (`claude-3-5-sonnet-20241022`).
* **Rationale**: 
  - Standardizes the pipeline on the core requirement of the brief.
  - Retains low API operational cost (under $0.05 per pipeline run) to satisfy the strict $10-$15 USD overall budget constraint.
  - Leverages Claude's superior instruction-following, structured extraction, and verification capabilities.

### 2.2 Orchestration: CLI wrapper (Future Trigger.dev Recommendation)
* **Decision**: For the MVP, we chose a code-first Python CLI orchestration script (`app/main.py`) executed via `infisical`.
* **Rationale**: 
  - Minimizes moving parts and deployment complexity for grading. 
  - Allows the entire pipeline to be run end-to-end with a single CLI command (`venv\Scripts\python.exe -m app.main`).
* **Production Recommendation**: We recommend **Trigger.dev** over **n8n**. Because our pipeline is code-heavy (parsing PDFs, custom Pydantic schema validation, and retry policies), a code-first orchestration framework like Trigger.dev fits better than n8n's visual node UI. It ensures type safety, version control, and seamless deployment as serverless background tasks.

### 2.3 Prompt Design & Schema Validation
* **Decision**: We use **Pydantic** for schema validation in Stage 3 and strict structural JSON formats in prompt instructions.
* **Rationale**: 
  - Pydantic ensures the LLM's response matches our schema (`ExtractedDocument`) exactly before saving, throwing clear errors if any field is missing or malformed.
  - The verification stage utilizes a percentage-based score (0-100) based on clear dimensions (Accuracy, Completeness, Hallucination) to guarantee the output meets the target $>85\%$ threshold.

## 3. What We'd Build Next (Future Scope)

With more time, the next production stages would be:
1. **Trigger.dev Deployment**: Wrap `app/main.py` in a Trigger.dev job to run automatically when new PDFs are uploaded to a cloud storage bucket (e.g. AWS S3).
2. **Publishing Integration**: Integrate the Confluence API in Stage 4 to publish the generated Markdown summaries directly to an Atlassian Cloud knowledge base.
3. **OCR for Scanned PDFs**: Add a preprocessing step using Tesseract or Google Cloud Document AI to support scanned image PDFs.
