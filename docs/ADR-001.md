## 1. Context & Problem Statement

For my Document Intelligence Pipeline project, I need to build a functional Minimum Viable Product (MVP) that ingests a PDF document, extracts and structures its text, publishes a clean Markdown summary, and runs a self-verification check. 

The main engineering challenges I faced are:
1. **Orchestration**: How to coordinate the 5 pipeline stages (Fetch $\rightarrow$ Ingest $\rightarrow$ Structure $\rightarrow$ Publish $\rightarrow$ Verify) securely and reliably.
2. **Schema Validation**: How to guarantee that the Large Language Model (LLM) returns output matching a strict JSON structure without crashing downstream components.
3. **Verification Accuracy**: How to design a self-verification rubric that is critical and "self-honest" instead of just giving a fake 100% score.
4. **Hosting & Serverless Timeouts**: How to deploy a trigger UI on Vercel without hitting Vercel's strict 10-second timeout limit for serverless functions, given that PDF processing and LLM audits take about 2 minutes.

## 2. Decision Trade-offs & Justifications

### 2.1 Pipeline Orchestration: n8n vs. Trigger.dev
To coordinate the stages, I analyzed two popular automation frameworks: **n8n** (a visual, node-based workflow builder) and **Trigger.dev** (a code-first, developer-focused background runner).

#### n8n (No-Code/Low-Code Visual Workflows)
* **Pros**: Very fast to connect APIs visually (e.g. download PDF $\rightarrow$ send to Slack). Great drag-and-drop dashboard.
* **Cons**: Running custom Python logic (like PyMuPDF parsing and Pydantic validation) inside visual nodes is messy. The workflows are stored as huge JSON blobs which are hard to track in Git, review in Pull Requests, or write unit tests for.

#### Trigger.dev (Code-First Background Jobs)
* **Pros**: Built entirely in code (TypeScript). Tasks are version-controlled in the repository, type-safe, and testable. It handles background queues and supports long-running jobs (up to 5–15 minutes) seamlessly.
* **Cons**: Requires a Node.js runtime and has a slightly higher initial learning curve compared to visual builders.

#### My Decision & Rationale:
I chose a **Hybrid Orchestration approach using Trigger.dev (for the web trigger) and a Python CLI script (for the local runner)**.
* **Why not n8n?** Our pipeline is code-heavy. We need custom chunking, precise PDF parsing, and strict Pydantic schema validation. Managing these in a visual UI like n8n leads to "spaghetti code." Trigger.dev allows me to write clean code that lives in my Git repository.
* **How the Hybrid approach helps**: 
  * I wrote [app/main.py](file:///app/main.py) to run the 5 stages sequentially using a single terminal command. This makes it extremely easy for a grader to clone and run the project locally.
  * I wrapped this script inside a **Trigger.dev background task** ([trigger/pipeline.ts](file:///trigger/pipeline.ts)). When triggered via the Vercel web page, the job runs as a background task. This completely bypasses Vercel's 10-second timeout limit and prints the live execution logs on the Trigger.dev dashboard.

### 2.4 Trigger.dev V3 Python Bundling & Directory Creation
During cloud deployment, we encountered two critical execution bugs:
1. **ModuleNotFoundError (`app.main`)**: By default, the Trigger.dev V3 compiler only packages Javascript/Typescript files referenced by task entry points. It completely excludes other source directories (like the Python `app/` package). We solved this by adding `scripts: ["app/**/*.py"]` under the `pythonExtension` builder configuration in `trigger.config.ts` to explicitly bundle the Python codebase.
2. **FileNotFoundError (`sample_data/sample.pdf`)**: In Stage 01 (Fetch), directory creation logic was only present in the local file upload branch. Running a remote PDF URL download failed because the target folder `sample_data` was never initialized. We resolved this by refactoring [app/main.py](file:///app/main.py) to run `os.makedirs` unconditionally at the start of the Stage 01 try block.

### 2.2 Prompt Design & Stage 03 Schema Validation
LLMs are probabilistic and return strings of text, which means they can output malformed JSON or miss fields. 

#### My Decision & Rationale:
I implemented strict **Pydantic validation** in [app/extraction/schema.py](file:///app/extraction/schema.py) coupled with an XML-delimited prompt structure.
1. **Strict System Prompt**: I provided Claude with the exact JSON Schema inside the system prompt and instructed it to return *only* the raw JSON, avoiding conversational wrappers.
2. **Pydantic Validation**: When the LLM returns the JSON, the code attempts to instantiate the Pydantic model `ExtractedDocument(**parsed)`. Pydantic checks all field types and lists at runtime. If the LLM misses a required field or returns a string instead of a list, Pydantic raises an immediate `ValidationError` rather than passing corrupt data downstream.

### 2.3 Verification Rubric & Self-Honesty (Stage 05)
A simple verification prompt often results in an inflated "100/100" score because LLMs tend to be agreeable and lenient.

#### My Decision & Rationale:
I designed a strict, objective verification prompt in [app/verification/verification_prompt.py](file:///app/verification/verification_prompt.py) based on a 100-point rubric (Accuracy: 50 pts, Completeness: 30 pts, No Hallucinations: 20 pts).
1. **Verbatim Quote Verification**: The auditor must check if the `evidence` field in the extraction exists *word-for-word* in the source document. If a quote is paraphrased or fabricated, the auditor deducts **10 points** under Accuracy.
2. **TOC Hallucination Check**: If the extractor lists headers from the Table of Contents as extracted claims, but the source text chunk does not actually explain those sections, the auditor deducts **5 points** under Hallucinations.
3. **Brutal Honesty**: I instructed the LLM auditor to act as a strict critic, penalizing every mismatch. In tests with simulated bad data (fake dates, paraphrased quotes), this verifier successfully downgraded the score, proving its reliability.

## 3. What I Would Build Next (Future Scope)

With more time, I would expand this pipeline into a production-grade enterprise application:

1. **Decoupled API Deployment**:
   * Currently, the Trigger.dev worker calls the Python script as a local subprocess. In production, I would deploy the Python code as a **FastAPI** web service hosted on **Railway** or **Render**. The Trigger.dev worker would then make a clean HTTP POST request to that API, removing the need for a local terminal.
2. **Direct Confluence Integration**:
   * Integrate the Confluence API to publish the generated Markdown summaries directly to an Atlassian Cloud knowledge base.
3. **OCR for Scanned PDFs**:
   * Add a preprocessing step using `pytesseract` or AWS Textract to support scanned image PDFs, as PyMuPDF only parses text-native PDFs.
4. **Enhanced Retry Policies & Dead Letter Queues (DLQ)**:
   * Configure Trigger.dev queue settings to automatically catch, alert, and store failed runs in a DLQ for developer inspection.
